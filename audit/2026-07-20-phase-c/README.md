## 調査3 既存の作業推奨機能

| # | 名前 | 場所 | 種類 |
|---|---|---|---|
| A | `globalNextTask`（👉 次にやる1件） | `src/App.jsx:13520-13581` | `WorkExecutionModal` 内の `useMemo` |
| B | `nextOptimalMove`（データ由来の次の一手） | `src/App.jsx:7147-7224` | **module直下の純関数** |
| C | `liveParallelGuide`（自動中に他の台でできること） | `src/App.jsx:13605-13673` | `useMemo`・`Date.now()` を直接使用 |

- 候補の条件: A=段取り工程優先→手動が動いていたら出さない→自動が動いている台は飛ばす→台優先×テンプレ順の最初の `waiting/paused`。
  B=`strictModeRules[型式_テンプレ].optimalOrder.enabled` が ON の組み合わせだけ。まとめて開始は「過去3ロット以上かつ同時開始率70%以上」。
  C=`workResource` が自動と別、または `parallelSafe` の工程だけ。
- **推奨の結果はどこにも保存されていない**（`logs` / `improvements` / `strict_mode_history` を確認。`strict_mode_history` に入るのは管理者の承認操作だけ）。
- 過去データからの再現可否:
  - **A は再現できる**。読むのは `lot.steps` / `lot.tasks[].status` / `lot.quantity` だけで、`Date.now()` を使わない。
    `src/domain/nextTask.js` へ切り出せば Phase C から同じ関数を呼べる。⚠ただし `tasks[].status` は**現在の状態**しか無いので、
    過去の任意時刻の状態は各タスクの `firstStartTime`/`endTime` から組み立て直す必要がある。
  - **C は再現できない**（`Date.now()` と端末ごとの表示 ON/OFF に依存）。
  - **B は部分的**。当時 `optimalOrder` が ON だったか、当時の完了ロット集合が何だったかは復元が要る。
- したがって Phase C では「**次にできる仕事があったか（計画手待ちか、着手していないだけか）**」の区別は
  **A の再現でおおむね出せる**。出せないのは「その瞬間に画面が何を勧めていたか」だけ。

## この調査で分からないこと（今のデータの限界）

1. 区間の中の**どこで**手を動かしていたか（推定は幅しか無い）→ 重なり時間は上限値
2. 休憩・監視で拘束されていた時間の除外（`interruptions` はあるが今回は未集計）
3. 完了押し忘れの自動区間（16本・8時間超）を機械的に除けない
4. 「担当者不明」3,087/4,187 しか担当が入っていない → 作業者別は全量ではない
5. 一括自動の台ごとの実態（過去は台別 `duration` が按分値）

## B.1.1 以降のデータで正確になること

- `task.sessions` = 実際に働いた区間そのもの（停止を挟んでも正しく足す）→ 重なりが**上限でなく実測**になる
- `lot.machineRuns` = 機械の壁時計（一括5台10分を50分にしない）
- 担当交代の帰属（`source:'handoff'`）
- 押し忘れの検出（開いたままの segment が分かる）
